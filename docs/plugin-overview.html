Why This Exists

动机：为什么需要这个插件？

Without the plugin

AI 生成网页 → 设计师束手无策

AI 生成网页的能力已经非常成熟——Claude、ChatGPT、Gemini 都能产出结构良好的 HTML/CSS。 但设计师拿到这些网页后，无法进行二次设计：改布局、调间距、换颜色、加动效， 这些本该在 Figma 里自由完成的动作全部被锁死在代码里。

With the plugin

AI 生成网页 → HTML 导入 Figma → 自由创作

HTML 作为中间件，把 AI 的生成能力和设计师的工作流无缝连接。 AI 产出高质量的可视化，插件将其转换为 Figma 图层，设计师在熟悉的工具里接手， 像编辑任何设计稿一样自由修改每一个像素。

Figma 原生方案的问题

Figma 提供了官方的 MCP 和 Plugin API，但在实际使用中体验很差： 不允许动态创建组件、无法修改 component properties、复杂的 auto-layout 操作受限。 更致命的是，让 AI 直接操作 Figma API 创建设计稿，AI 难以理解空间关系、图层嵌套和约束系统， 出了错根本不知道是 AI 的问题还是 API 的问题。

而 HTML/CSS 是 AI 真正擅长的「图形化语言」——表达能力强，生成质量稳定，可以直接在浏览器里预览。 用 HTML 作为桥梁，比让 AI 直接学 Figma API 要可靠得多。

Real-world Use Case

实际使用场景：PRD → HTML → Figma

问题：设计师如何在长篇 PRD 中不迷路？

工作中拿到一份产品的 PRD，几十页的文档里混杂了给开发看的技术约束、给测试看的验收标准、 给运营看的 KPI 指标，以及真正给设计师看的交互逻辑、页面结构和信息层级。 设计师要在长篇文字描述中找到自己需要的内容，很容易信息过载。

AI 确实可以帮助从 PRD 中提取和筛选重点。但一旦描述复杂化——多状态、多分支、 嵌套的逻辑关系——纯文本的交付件无法让设计师快速理解信息架构。

"Markdown 不是给人类确认的合适交付件。真正需要人类确认的环节，还是需要图形化。"
— Andrej Karpathy 的观点

AI 最擅长的图形化语言就是 HTML

把 PRD 交给 AI 转换为 HTML 后，信息层级、模块关系、优先级差异一目了然： 用颜色区分重点与非重点，用卡片组织模块关系，用间距表达信息层级，用排版建立阅读节奏。 例如 Batch Reward 的 PRD，经过 AI 转换成 HTML 之后，设计师一眼就能看到需要做什么， 不再需要在文字中苦苦寻找设计线索。

Core Insight

跨 Frame 的 instanceof 陷阱

这是开发过程中遇到的最隐蔽的坑。HTML 内容在 iframe 中渲染，但提取逻辑从父窗口执行。 由于 iframe 和父窗口各自拥有独立的 JavaScript 原型链，所有 instanceof 检查全部失效。

失效写法（iframe 元素）跨 Frame 兼容写法
el instanceof SVGSVGElementel.tagName === 'svg'
el instanceof SVGElementel.namespaceURI === 'http://www.w3.org/2000/svg'
el instanceof HTMLImageElementel.tagName === 'IMG'
el instanceof HTMLPictureElementel.tagName === 'PICTURE'
el instanceof HTMLVideoElementel.tagName === 'VIDEO'
el instanceof HTMLElementel.nodeType === 1
el instanceof Elementel.nodeType === 1
ref instanceof HTMLElementref.nodeType === 1
Pitfalls

踩坑记录

⚠

CSP 拦截外部样式表

Figma 插件环境的 CSP 禁止加载外部 CSS。HTML 中的 Google Fonts 链接和 @import 规则需要预先去除。

html.replace(/<link[^>]*href=["']https?:\/\/[^"']*["'][^>]*\/?>/gi, '')
◈

内联元素的包围盒异常

当 inline 元素包含 block 子元素时，getBoundingClientRect 返回的宽度会偏大。需要计算子元素的聚合包围盒来修正。

◫

非统一边框无法直接映射

CSS 允许四边不同宽度/颜色的边框，但 Figma RECTANGLE 只支持统一边框。需要为每条边单独创建细长的 RECTANGLE 图层。

T

字体渲染差异导致文本溢出

CSS 字体和 Figma 内置字体的渲染结果可能存在差异。创建文本后需要检测溢出并逐步缩小字号（最多缩小 30%）来适配。

⊘

iframe 内脚本需隔离

HTML 中内联的 <script> 标签会在 iframe 中执行，可能抛出错误干扰提取流程。预览时替换 script 标签为注释。

F

字体回退链跨平台匹配

HTML 使用 "Inter, -apple-system, sans-serif" 回退链，但 Figma 字体有限。需要遍历回退链并在 Figma 可用字体中找到第一个匹配项。

Design Decisions

关键设计决策

决策选择原因
坐标体系绝对定位 (getBoundingClientRect)不重建 CSS 布局，100% 还原视觉效果，避免 flexbox/grid → auto-layout 的阻抗不匹配
图层结构扁平数组 + 可选树结构 (useFrames)快速导入用扁平模式，需要层级关系时启用 makeTree LCA 算法
提取位置iframe 内渲染，父窗口提取隔离 HTML 运行环境，防止污染 Figma 插件 UI 的 DOM
跨 Frame 检查tagName / nodeType / namespaceURIinstanceof 在跨 frame 场景下因独立原型链而失效
字体策略保留原始 CSS font-family 字符串解耦提取和渲染阶段，在主线程灵活回退匹配 Figma 可用字体
