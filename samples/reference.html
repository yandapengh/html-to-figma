02

社区方案：graphify 是如何工作的？

一个面向 AI 编码助手的技能，对已有目录运行，输出知识图谱报告 —— 事后代码考古。

100%
输出 God nodes · 意外连接 · 建议提问 边标记 EXTRACTED / INFERRED / AMBIGUOUS 无 embedding 纯图拓扑聚类
展开技术细节：两轮提取、关系类型与聚类原理

代码提取 完全确定性，使用 tree-sitter 解析 AST，生成调用图、继承关系、导入依赖。不消耗 LLM token。

文档 / 图片 并行调用 Claude 子代理，从 Markdown、PDF、截图中抽取概念，建立与代码实体的关联，边标记为 INFERRED 并附带置信度。

Leiden 聚类 基于图拓扑边密度进行社区划分，不依赖 embedding 或向量数据库。语义相似边直接参与聚类，因此概念相近的节点自然聚合。

关系类型 calls, imports, inherits_from, semantically_similar_to, rationale_for 等，均为描述性关系。

我们的方案 vs graphify —— 关键差异

从目标、处理时机到运行时产出 —— 七个维度的系统对比，揭示两种方案的本质区别。

维度 graphify 我们的数学化 Figma 代码库
目标 理解已有代码库结构 复用片段，自动组合生成新代码
处理时机 事后（对目录运行） 实时增量编译（对话中）
代码提取 tree-sitter AST 结构 LLM 语义抽象（模式+参数化）
核心关系 calls / imports / similar_to produces / consumes / variant_of
聚类依据 Leiden 边密度 功能兼容性 + 抽象层次 + 类型匹配
参数化 无 核心能力（颜色/尺寸/数量变体）
运行时输出 图路径解释、报告 可执行组合代码
本质差异 graphify = 事后理解 · 我们 = 实时合成 关系差异 描述性边 → 操作性接口边
案例解释：为什么 AST 不够？

两段语义等价的 Figma 代码：

// 写法 A
rect.fills = [{ type: 'SOLID', color: { r: 1, g: 0, b: 0 } }];

// 写法 B
const paint = figma.util.solidPaint(1, 0, 0);
rect.fills = [paint];

AST 将这两个视为完全不同的结构（属性赋值 vs 函数调用），但我们需要知道它们都是 设置红色填充 这一模式的变体。 LLM 抽象可以识别语义等价，并为其建立 variant_of 边，后续检索时两者都能被命中。

展开细节：每一层如何防止概念漂移？
锚定策略（Prompt 层）

每次增量抽象时，在 Prompt 中植入最相似的已有模式作为 Few-shot 示例。例如，如果新模式是一个"带阴影的卡片循环"，系统会先检索出"普通卡片循环"的模式图作为参考，要求 LLM 在此基础上增加阴影步骤，而不是重新创造描述。

硬校验（程序层）

- JSON Schema 验证结构
- 接口类型匹配（produces/consumes 类型必须与已有节点定义一致）
- 不合格输出自动打回，要求 LLM 修正（最多重试 2 次），零 token 消耗。

拓扑监控（结构层）

入库后运行社区发现算法，若新模式导致原有社区分裂，或成为意外的 God node，触发预警。同时计算新模式与声称相似的已有模式之间的图距离，距离过远说明抽象不一致。

编译日志

每次增量记录：片段 hash、抽象结果 JSON、引用的已有模式、是否引入新节点、校验结果。支持回溯和定向修复。
